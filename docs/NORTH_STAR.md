## Product Thesis — почему рынок устроен неправильно

> **Поиск клиентов — единственная бизнес-задача, которую люди до сих пор решают так, как решали 20 лет назад: вручную, медленно, случайно.**

Каждый день миллионы людей открывают браузер и начинают искать клиентов: перебирают справочники, листают HH.ru, пишут в личку незнакомцам — без системы, без данных, без понимания, кто из них реально готов к разговору.

Они тратят часы на то, что машина может сделать за минуты.

Это не потому что людям не нужно лучше. Это потому что все инструменты, которые существуют — Apollo, Clay, HubSpot — построены для другой аудитории: для профессиональных sales-команд с бюджетами, обучением и выделенными data-ops специалистами.

**Мы убеждены:** первый продукт, который принимает намерение на естественном языке и сам делает всё остальное — изменит то, как B2B-продажи работают для большинства.

## Emotional Promise — что чувствует пользователь

Через неделю использования пользователь должен почувствовать одно:

> **«Я больше не сижу без заказов — я знаю, что делать, и делаю это».**

Конкретнее:
- Утром открываешь продукт — и уже знаешь, кому писать сегодня.
- Не тратишь время на поиск: кандидаты уже есть, контакты уже есть.
- Не мучаешься с первым сообщением: черновик уже готов, остаётся только прочитать.
- Нет ощущения «я трачу время на CRM» — есть ощущение «я занимаюсь продажами».

Это разница между продуктом, который помогает работать, и продуктом, который **заменяет ту часть работы, которую люди ненавидят больше всего**.

## Стратегический выбор аудитории

**Мы строим продукт для любых B2B-продавцов — не только для профессиональных sales-команд.**

Это означает:
- Sales-терминология (SDR, GTM, Pipeline, Sequences, Activity Queue, ICP) существует **только** во внутренних документах, архитектуре и API. В пользовательском интерфейсе её нет.
- Пользователь — это любой человек, который что-то продаёт другому бизнесу.
- Если продукт понятен юристу, который никогда не работал с CRM — он точно будет понятен опытному SDR.

**Почему Вариант A, а не B:**  
Профессиональных SDR в России — десятки тысяч. Людей, которые продают в B2B не называя это «продажами», — миллионы. Вариант B (только для sales-команд) — это конкуренция с Apollo на его поле. Вариант A — это свободный рынок.

## Почему этот продукт существует

Есть огромное количество людей, которые что-то продают другому бизнесу.

Юристы. Фотографы. Дизайнеры. Разработчики. Перевозчики. Строители. Консультанты. Производители. Маркетинговые агентства. Поставщики оборудования. Дистрибьюторы.

У всех у них одна и та же проблема:

**Они не знают, где найти следующего клиента.**

Или знают — но на поиск уходят часы, дни, а иногда нанимают отдельного человека.

Инструменты, которые существуют сегодня, не решают эту проблему для большинства из них. Apollo — для продвинутых sales-команд. HubSpot — для корпораций с бюджетом. Clay — для data-ops специалистов. Все они требуют изучения, настройки, и только потом дают результат.

**Мы строим другое.**

Продукт, который работает с первой минуты. Для любого человека, который хочет найти клиентов — независимо от опыта, профессии и размера команды.

Это то, что не строит никто. Не потому что не могут — потому что не думают в этом направлении.

## Философия: Intent-First

Пользователь никогда не мыслит терминами системы.

Он не создаёт «Hunt». Он не настраивает «Waterfall Enrichment». Он не строит «ICP-профиль».

Он думает:

> *«Мне нужны транспортные компании в Екатеринбурге»*  
> *«Мне нужны рестораны, которые ещё не работают с доставкой»*  
> *«Мне нужны юридические фирмы, которые недавно открылись»*

Это **намерение**. Вся система строится вокруг того, чтобы принять это намерение и превратить его в готовые контакты — с минимальным участием пользователя.

## Два слоя продукта

Это одно из ключевых архитектурных решений. Оба слоя существуют, но только один из них видит пользователь.

### Пользовательский слой (что видит пользователь)

| Что пользователь делает | Что пользователь видит |
|------------------------|----------------------|
| Описывает кого ищет | Результаты поиска |
| Выбирает подходящих | Контакты с информацией |
| Пишет первым | Персональное сообщение |
| Получает ответы | Входящие сообщения |
| Продолжает общение | История переписки |

Язык пользовательского слоя: поиск, список клиентов, контакты, сообщения, история, следующий шаг.

### Системный слой (что пользователь никогда не видит)

Hunt · Waterfall Enrichment · ICP Score · Signal Engine · Enrichment Orchestrator · CRM · Pipeline · Sequences · Activity Queue · Events · AI Scoring · Plugin Registry

Это внутренние механики. Они живут в API, базе данных, внутренних документах. Но никогда — в пользовательском интерфейсе.

**Правило:** если пользователю нужно знать слово из системного слоя, чтобы пользоваться продуктом — что-то пошло не так.

## UX-принципы (неизменяемые)

Эти принципы — ограничения, а не рекомендации. Любая функция, нарушающая их, не попадает в продукт.

### 1. Zero Learning Curve
Пользователь, открывший систему впервые, понимает что делать без обучения, туториалов и онбординг-видео. Интерфейс объясняет себя.

### 2. Zero Cognitive Load
Система никогда не ставит пользователя перед выбором, который он не хочет делать:
- Никогда: «выбери источник данных»
- Никогда: «настрой параметры поиска»
- Никогда: «выбери следующий шаг»

Система выбирает сама. Пользователь может изменить — но не обязан.

### 3. Intent → Result
Каждый сценарий начинается с намерения и заканчивается результатом. Всё между ними — автоматика.

### 4. Пустой экран — это провал
Новый пользователь должен видеть поиск клиентов, а не пустую очередь задач. «У вас пока нет задач» — худший первый опыт. Система должна немедленно предложить начать поиск.

### 5. Минимум экранов
Если задачу можно решить на одном экране — она решается на одном. Новый экран создаётся только при крайней необходимости.

### 6. Автоматика по умолчанию, контроль по желанию
Система работает автоматически. Пользователь может вмешаться — но не обязан. Умные defaults важнее гибкой настройки.

### 7. AI предлагает — человек решает
AI автоматизирует, подготавливает и предлагает. Не «думает вместо» и не «знает лучше». Критичные действия всегда требуют подтверждения пользователя. Human-in-the-loop — это не ограничение, это принцип.

## Конкурентное преимущество

### Главное: Мы не просто проще — мы лучше для нашей аудитории

Мы проще Apollo. Проще Clay. Проще HubSpot. Но простота — это не наше единственное оружие.

Apollo не просто сложный — он **бесполезен** для юриста, который ищет корпоративных клиентов в России. Не потому что сложен, а потому что не знает ни 2GIS, ни HH.ru, ни российской деловой культуры. Clay не просто требует обучения — он требует **отдельного специалиста** для работы с ним. Close — лучшая CRM для телефонных продаж, но не ищет клиентов вообще.

Наша аудитория — те, для кого эти инструменты **не работают принципиально**, не просто «работают хуже».

**Мы лучше, потому что:**
- **Находим клиентов там, где конкуренты слепы** — российские источники как встроенная часть, не надстройка
- **Понимаем намерение, а не параметры** — пользователь пишет как думает, не как устроена база данных
- **Дают результат без настройки** — не «можно получить результат», а «результат по умолчанию»

Это не «меньше возможностей». Это «возможности скрыты под капотом». Мощь есть — пользователь её не видит.

### Усилители (важные, но не определяющие):

**Российские источники данных**  
2GIS, HH.ru, Dadata, Rusprofile, Kontур.Фокус — ни один западный конкурент их не поддерживает. Лучшее покрытие для России и СНГ.

**Telegram как канал**

## Moat — почему нас трудно повторить

Простоту скопировать легко. Интеграцию с 2GIS — тоже. Это не настоящий ров.

**Настоящий защитный ров — данные, которые накапливаются с каждым действием пользователя.**

### Data Flywheel

Каждый раз, когда пользователь ищет клиентов, система учится:
- Какие компании в этой отрасли чаще соглашаются на разговор
- Какие сигналы («открыл новый офис», «нанимает менеджеров») предшествуют сделке
- Какой тип первого сообщения получает ответ, а какой — нет

Это знание не исчезает. Оно накапливается:

```
1 000 пользователей → базовые паттерны по отраслям
10 000 пользователей → сигналы готовности к покупке
100 000 пользователей → предиктивная модель: «эта компания, скорее всего, готова»
```

**Через 3 года наш продукт знает о том, как устроены B2B-продажи в России, больше, чем любой конкурент — не потому что у нас лучшие алгоритмы, а потому что у нас данные из реальных сделок.**

### Network Effects

Один пользователь, который ищет строительные компании в Екатеринбурге, создаёт данные, которые делают поиск лучше для следующего.

Не потому что они делятся данными между собой. Потому что система накапливает понимание рынка.

> **Чем больше пользователей — тем умнее продукт. Это не маркетинг — это механика.**

### Почему большие игроки не скопируют это быстро

OpenAI, Anthropic, Google могут построить поиск на естественном языке. Но:
- У них нет интеграций с 2GIS, HH.ru, Rusprofile и другими российскими источниками
- У них нет данных о том, кто реально отвечает на сообщения в российском B2B
- Они не будут строить специализированный продукт для рынка, который считают нишевым
- Их реакция на «российский B2B intent graph» занимает годы, не месяцы

**Наш ров — не технология. Наш ров — специализация + данные + время.**

Любой сильный стартап отвечает на вопрос: *«Почему вы, а не кто-то другой?»*

## Заметка о названии

*Внутренняя заметка для команды, не для пользователей.*

Текущее название «AI Sales OS» создаёт противоречие, которое будет нарастать.

Документ говорит: *не sales, не OS, не CRM, не GTM*.  
Название говорит: *именно это*.

Первое слово, которое видит новый пользователь — «Sales» — как раз то, от чего мы уходим.

**Мы не меняем название прямо сейчас.** Смена бренда — дорогостоящее решение, и его нужно принимать осознанно. Но это противоречие нужно зафиксировать сейчас, чтобы не строить на нём маркетинг, лендинги и позиционирование.

Направление для размышления: продукт можно описать как *«сервис, который находит вам клиентов»* — без слова «sales», без слова «OS». Хорошее название должно звучать как то, чем продукт является, а не как то, чем он конкурирует.

### Чем мы не являемся:
- Не «Apollo для России» — российские данные лишь усиливают продукт
- Не «CRM нового поколения» — CRM есть, но это не продукт
- Не «GTM-платформа» — это язык инвесторов, не пользователей

## Принципы принятия решений

### Главный вопрос для любой функции:

> **«Если убрать эту функцию — пользователь всё ещё сможет найти клиента и написать ему за 10 минут?»**

Если **да** — функция не является ядром. Она может быть полезной, но не попадает в MVP.

### Дополнительные вопросы:

**«Может ли человек без опыта в продажах понять это за 30 секунд?»**  
Если нет — упрощаем или убираем.

**«Можно ли сделать это автоматически вместо пользователя?»**  
Если да — делаем автоматически.

**«Требует ли это знания слова из системного слоя?»**  
Если да — переосмысляем.

**«Добавляет ли это экран, настройку или выбор?»**  
Если да — сначала докажи, что без этого нельзя.

## Почему пользователь будет возвращаться каждый день

Продукт создаёт привычку через ежедневный ритуал.

```
Утро → открываешь систему
     → видишь кому написать сегодня и почему
     → пишешь → следующий → следующий
     → к концу дня — список сделан
```

Это не «CRM, где надо вести записи». Это «ежедневный помощник, который каждый день говорит что делать дальше».

Пустой список — только у нового пользователя. После первого поиска система всегда знает что предложить.

## Продуктовые ограничения (что никогда не делаем)

| Ограничение | Причина |
|------------|---------|
| Sales-терминология в UI | Нарушает Zero Learning Curve |
| Конфигуратор источников данных | Пользователь не должен выбирать провайдеров |
| Встроенный командный чат | Дублирует Telegram/Slack |
| Геймификация и leaderboards | Не работает в B2B |
| AI-прогноз выручки (до 6 мес. данных) | Нет данных = нет точности |
| Маркетинговые email-кампании | Другой продукт |
| Live-chat с клиентами | Другой продукт |
| Billing/Invoice management | Другой продукт |

## Критерии оценки любой новой функции

Функция проходит отбор, если отвечает «да» на большинство вопросов:

- [ ] Без этой функции пользователь НЕ может найти клиента и написать ему за 10 минут?
- [ ] Может ли новый пользователь понять это без объяснений?
- [ ] Делает ли это работу автоматически вместо пользователя?
- [ ] Убирает ли это необходимость думать о следующем шаге?
- [ ] Нужно ли это > 50% наших пользователей каждый день?
- [ ] Нет ли уже лучшего инструмента для этого (Slack, DocuSign и т.д.)?
