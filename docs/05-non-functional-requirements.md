# Non-Functional Requirements — AI Sales OS

## NFR-01. Производительность

| Требование | Целевое значение | Метод измерения |
|-----------|-----------------|----------------|
| Время ответа API (p95) | < 300 ms | APM (Sentry/Datadog) |
| Время загрузки дашборда | < 1.5 s (LCP) | Lighthouse CI |
| Пропускная способность | 10,000 лидов/день без деградации | Нагрузочный тест (k6) |
| Время обогащения одного лида | < 30 сек (waterfall) | Queue metrics |
| Время генерации письма (AI) | < 5 сек | API latency log |
| Объём БД без деградации | до 10 млн лидов | PostgreSQL benchmarks |

### Почему так:
- 300 ms — стандарт отзывчивого SaaS по Google UX research
- 10,000 лидов/день — потолок активной кампании для клиента с 100 SDR
- 30 сек на обогащение — допустимо для фонового процесса, не блокирует UI

## NFR-02. Надёжность и доступность

| Требование | Значение |
|-----------|---------|
| Uptime SLA | 99.5% (допустимо ~3.6 ч простоя/мес) |
| Потеря задач в очереди | 0% (персистентная очередь) |
| RTO (Recovery Time Objective) | < 15 минут |
| RPO (Recovery Point Objective) | < 1 час |
| Деградация при сбое AI-провайдера | Fallback на резервную модель |

### Почему 99.5%, а не 99.9%:
MVP-стадия. 99.9% требует PgBouncer, multi-AZ, геобалансировки — это капитальные затраты. 99.5% достигается single-AZ + readreplica + мониторинг, стартуем с этого.

---

## NFR-03. Масштабируемость

- **Горизонтальное масштабирование**: все воркеры (enrichment, email, AI) stateless — масштабируются добавлением инстансов
- **Мультитенантность**: данные изолированы на уровне `workspace_id` в PostgreSQL (row-level security)
- **Очередь**: BullMQ/Redis масштабируется через Redis Cluster при нагрузке > 100K задач/день
- **Хранение вложений**: S3-совместимое хранилище (не PostgreSQL BLOB)
- **Целевой потолок v1**: 500 workspace, каждый до 100K лидов → 50M строк в БД

---

## NFR-04. Безопасность

- **Аутентификация**: JWT + Refresh Token, срок жизни access token — 15 минут
- **Авторизация**: RBAC (роли: owner, admin, manager, sdr) + проверка workspace_id на каждом запросе
- **Шифрование at rest**: AES-256 для чувствительных полей (email, телефон, ИНН)
- **Шифрование in transit**: TLS 1.3 обязателен
- **API ключи внешних сервисов**: хранятся в зашифрованном виде, не выводятся в логи
- **Rate limiting**: 1,000 req/мин/IP на public endpoints; 10,000 req/мин на authenticated
- **Audit log**: каждое изменение данных пишется в append-only лог с user_id и timestamp
- **OWASP Top 10**: защита от SQLi, XSS, CSRF, IDOR, mass assignment

---

## NFR-05. Соответствие законодательству

- **152-ФЗ (персональные данные РФ)**: хранение ПД только с согласия или законным основанием; возможность удаления по запросу субъекта
- **GDPR** (если клиенты из ЕС): право на забвение, DPA-соглашения, ограничение трансграничной передачи
- **CAN-SPAM / Закон о рекламе (РФ)**: обязательная ссылка отписки, идентификатор отправителя
- **Avito/LinkedIn ToS**: не парсим напрямую, используем официальные API или легальных агрегаторов

---

## NFR-06. Наблюдаемость (Observability)

- **Логирование**: структурированные JSON-логи; уровни: DEBUG/INFO/WARN/ERROR; трассировочный request_id
- **Метрики**: Prometheus + Grafana или встроенный дашборд; ключевые метрики: queue depth, worker latency, API error rate, AI token usage
- **Трейсинг**: OpenTelemetry; end-to-end trace для каждого лида (generation → enrichment → sequence → send)
- **Алерты**: email/Telegram при error rate > 1% за 5 минут или очередь > 10,000 необработанных задач
- **AI наблюдаемость**: каждый запрос к LLM логируется с prompt, response, токенами, стоимостью, latency

---

## NFR-07. Тестируемость

- **Unit coverage**: > 80% для бизнес-логики (enrichment, scoring, sequence engine)
- **Integration tests**: для всех API-интеграций (с моками внешних сервисов)
- **E2E tests**: для критических user journeys (создание кампании → отправка → трекинг)
- **Load tests**: k6 сценарии для throughput validation
- **AI tests**: golden dataset для оценки качества генерации писем (человеческая оценка + автоматическая)

---

## NFR-08. Поддерживаемость

- **Монорепозиторий**: все сервисы в одном repo с чёткими границами модулей
- **Документированный API**: OpenAPI 3.1 spec автогенерируется из кода
- **Миграции БД**: только через Drizzle ORM migrations, без ручного SQL в prod
- **Feature flags**: для безопасного раскатывания новых функций без деплоя
- **Onboarding нового разработчика**: `docker-compose up` → рабочая среда за < 10 минут

---

## NFR-09. Интернационализация

- **Язык интерфейса**: русский (основной), английский (опционально в v2)
- **Часовые пояса**: все timestamps хранятся в UTC, конвертируются на клиенте
- **Валюта**: рубли (основная), USD (для международных клиентов)
- **Телефонные форматы**: поддержка +7 и международных форматов
