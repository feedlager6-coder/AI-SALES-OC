## Важное решение, принятое перед написанием этого документа

**Мы строим продукт для любых B2B-продавцов — не только для профессиональных sales-команд.**

Это означает одно конкретное следствие для всего документа:

- **Пользовательский слой** — язык, понятный юристу, который никогда не слышал слово «CRM»
- **Системный слой** — архитектурные термины (Hunt, ICP, Pipeline, Waterfall, Activity Queue), которые пользователь никогда не видит

Там, где в предыдущих версиях стоял термин из системного слоя — он теперь либо переведён в пользовательский язык, либо явно помечен как внутренний.

## Вопрос 1. Философия AI Sales OS

### Одна идея

Пользователь приходит не за CRM.  
Пользователь приходит **получить новых клиентов**.

Всё остальное — следствие этого.

Когда юрист открывает наш продукт, он не думает «мне нужна CRM» — он думает «мне нужны корпоративные клиенты». Когда перевозчик открывает — «мне нужны грузоотправители». Когда дизайнер — «мне нужны заказы от компаний».

Продукт существует ровно для одной цели: **превратить это намерение в готовые контакты — быстро, без усилий, без обучения.**

База данных клиентов появляется сама. История переписки ведётся сама. Следующий шаг предлагается сам. Пользователь делает только то, что может сделать только человек: принимает решение.

### Три принципа

**1. Намерение → Результат, минуя всё лишнее**  
Пользователь говорит кого ищет — система делает остальное. Нет настроек провайдеров. Нет построения воронки. Нет выбора шаблонов из 47 вариантов.

**2. AI подготавливает и предлагает — человек решает**  
AI не «думает вместо» и не «знает лучше». AI автоматизирует рутину: готовит информацию перед звонком, предлагает заметку после разговора, составляет черновик первого сообщения. Пользователь одобряет или меняет — но никогда не начинает с чистого листа.

**3. Простота — это конкурентное оружие, а не компромисс**  
Главная проблема конкурентов — не плохие данные. Главная проблема — когнитивная перегрузка. Apollo встречает нового пользователя 150 параметрами поиска. Мы встречаем его одним полем: «Кого вы ищете?»

## Вопрос 2. Что является продуктом?

Если объяснить в одном предложении, понятном любому:

> **AI Sales OS — это сервис, который помогает найти потенциальных клиентов и написать им первое сообщение.**

Внутри это сложная система: поиск по десяткам источников, каскадное обогащение данных, AI-приоритизация, автоматические персонализированные сообщения. Но пользователь этого не видит.

Он видит только: описал кого ищет → получил список с контактами → написал первым.

**Внутреннее название для команды:** GTM Intelligence Platform.  
**Название для пользователя:** просто AI Sales OS — сервис для поиска клиентов.  
**Что это точно не:** CRM, Apollo-клон, Sales OS (GTM — язык инвесторов, не пользователей).

## Вопрос 3. Главная сущность системы

### Пользовательская сущность: Намерение

Пользователь не создаёт «Hunt». Не настраивает «поисковый профиль». Не строит «ICP».

Он описывает что хочет:

> *«Мне нужны транспортные компании в Екатеринбурге»*  
> *«Мне нужны рестораны, которые недавно открылись»*  
> *«Мне нужны юридические фирмы с корпоративными клиентами»*

Это его единица работы. В интерфейсе это называется «поиск» или «список клиентов» — не «Hunt», не «ICP-профиль».

### Системная иерархия (пользователь этого не видит)

```
[Пользователь вводит намерение]
         ↓
    Hunt (внутр.)        ← создаётся системой автоматически
         ↓
    Company              ← то, что пользователь видит
    ├── Contact
    ├── History          ← вся история общения (внутри: Timeline + Events)
    └── Next Step        ← что сделать дальше (внутри: Activity Queue)
```

**Правило:** слово «Hunt» существует в API (`/api/v1/hunts`), в базе данных (таблица `hunts`), во внутренней документации. В интерфейсе пользователя — никогда.

### Почему Company остаётся бизнес-сущностью

В B2B деньги платит организация. Контакт — точка входа, но решение принимает компания. Это решение из v1.0 остаётся неизменным.

## Вопрос 4. Путь пользователя с первой секунды

### Цель: отправить первое персонализированное сообщение релевантному клиенту за 10 минут

Это North Star Metric продукта.

### Онбординг

```
Регистрация (30 секунд)
  → только email + пароль
  → никаких вопросов о размере команды и текущих инструментах

Экран 1: [новый пользователь]
  Не пустая очередь задач.
  Не пустой список компаний.
  
  Сразу:
  «Кого вы ищете? Опишите своими словами»
  [___________________________]
  
  Пример: «Строительные компании в Москве, 20–100 человек»

── Interactive Intent (ключевой момент доверия) ──

  Прежде чем запустить поиск, система показывает:
  
  ┌────────────────────────────────────────────┐
  │ Вот как я понял ваш запрос:                │
  │                                            │
  │ ✅ Отрасль: строительство                  │
  │ ✅ Регион: Москва                           │
  │ ✅ Размер: 20–100 сотрудников              │
  │ ❓ Вас интересуют только новые компании?   │
  │    [Да, новые] [Нет, любые]               │
  │                                            │
  │ [Всё верно, искать] [Уточнить]            │
  └────────────────────────────────────────────┘
  
  Это не дополнительный шаг — это момент, когда пользователь
  понимает: система меня слышит и поняла правильно.

Экран 2: Результаты поиска (через ~2 минуты)
  ┌────────────────────────────────────────┐
  │ ТехноТранс                             │
  │ Логистика · Екатеринбург               │
  │ Иван Петров (руководитель)             │
  │ ivan@technot.ru · +7 (343) ...         │
  │ 📌 Недавно открыли 3 вакансии          │
  │ [Написать] [Позвонить] [Пропустить]   │
  └────────────────────────────────────────┘

Экран 3: Первое сообщение готово
  AI подготовил персональное письмо.
  Пользователь читает → [Отправить] или [Изменить]
  
  Готово. Первый потенциальный клиент — в работе.
```

**Что здесь никогда не происходит:**
- Нет «выберите провайдера данных»
- Нет «настройте воронку»
- Нет «импортируйте CSV для начала»
- Нет «у вас пока нет задач»

### Почему Interactive Intent критичен для доверия

Пользователь видит «магию» — описал запрос, и через 2 минуты появился список. Но магия без прозрачности порождает недоверие:

*«Система нашла рестораны — а я просил кафе. Что она вообще ищет?»*

Interactive Intent разрывает этот страх в зародыше. Прежде чем начать поиск, система переводит запрос в читаемые параметры и спрашивает: «Я правильно понял?»

Это не технический шаг. Это **психологический контракт**: пользователь видит, что система думает прозрачно, и начинает ей доверять.

**Правила для Interactive Intent:**
- Не более 3–4 параметров в подтверждении (не перечислять всё)
- Показывать только то, что система добавила сама (не то, что явно сказал пользователь)
- Задавать максимум 1 уточняющий вопрос
- Всегда давать выход «Всё верно, искать» — без обязательного уточнения

## Вопрос 5. Путь пользователя, который хочет найти клиентов

Сценарий одинаковый для юриста, фотографа и руководителя отдела продаж.

**Пользователь:** «Мне нужны юридические фирмы в Москве, которые работают с корпоративными клиентами»

**Что делает система (всё невидимо для пользователя):**

1. Интерпретирует намерение: отрасль = юридические услуги, регион = Москва, признак = корпоративные клиенты
2. Ищет в 2GIS, HH.ru, Rusprofile, Google Business
3. Определяет «работает с корпоративными клиентами» через вакансии и описания
4. Находит контакты руководителей через каскад источников: Dadata → Hunter → Snov → сайт
5. Расставляет приоритеты: выше те, кто больше похож на успешных клиентов
6. Отмечает сигналы: «открыла новый офис», «нанимает юристов»

**Что видит пользователь через 2–3 минуты:**  
Готовый список из 50 компаний. Контакты. Краткие пометки почему каждая интересна. Три примера первого сообщения под разные типы компаний.

Пользователь нигде не выбирал источники. Нигде не настраивал каскадное обогащение. Нигде не думал про приоритизацию. Система сделала всё сама.

## Вопрос 6. Ментальная модель пользователя

### Как пользователь должен думать о системе:

**Не:** «Это CRM, где я веду клиентов»  
**Не:** «Это платформа GTM-автоматизации»  
**Да:** «Это сервис, который помогает мне находить клиентов и знает что делать дальше»

### Три режима, понятных без инструкций:

**«Найти» — когда нужны новые клиенты**  
Описал кого ищешь → система нашла → выбрал → написал

**«Продолжить» — когда уже есть с кем работать**  
Открываешь систему → видишь «кому написать сегодня и почему» → делаешь  
*(внутри: Activity Queue, Pipeline — пользователь этих слов не видит)*

**«Посмотреть» — когда нужно понять что происходит**  
Сколько людей ждут ответа. Кто ответил. Что работает.

## Вопрос 7. Какие экраны нужны

### Правило: каждый экран должен отвечать на вопрос «что мне здесь сделать?»

**9 экранов — максимум для MVP:**

| Экран (пользовательское название) | Вопрос, на который отвечает | Внутренний термин |
|----------------------------------|----------------------------|-------------------|
| **Главный / Поиск** (для новых) | «С чего начать?» | — |
| **Сегодня** (когда есть данные) | «Что мне сделать прямо сейчас?» | Activity Queue |
| **Мои клиенты** | «Кто у меня уже есть?» | Companies |
| **Карточка компании** | «Что я знаю об этой компании?» | Company Card + Timeline |
| **Контакт** | «Кто это и как с ним связаться?» | Contact Card |
| **Воронка** | «На каком этапе каждый клиент?» | Pipeline (Kanban) |
| **Рассылки** | «Какие письма я отправил?» | Sequences |
| **Входящие** | «Кто мне ответил?» | Inbox |
| **Настройки** | «Мой профиль и подключения» | Settings |

**Главное правило для первого экрана:**
- Новый пользователь (нет данных) → видит поиск клиентов
- Вернувшийся пользователь (есть данные) → видит «что делать сегодня»

### Экранов нет и не будет:

| Экран | Причина |
|-------|---------|
| Провайдеры данных / настройка источников | Пользователь не выбирает источники |
| Конфигуратор ICP | ICP формируется через диалог |
| Аналитика (отдельный раздел) | 5 метрик прямо там, где нужны |
| Конструктор отчётов | 95% не откроют |
| Карта контактов | Не помогает продавать |
| Управление плагинами | Внутренняя механика |

## Вопрос 8. Что перестаёт быть важным

**Ручное создание компаний и контактов**  
Компании появляются как результат поиска. Ручной ввод — крайний случай, не основной путь.

**Импорт CSV как точка входа**  
Остаётся как опция. Перестаёт быть «с чего начать».

**Выбор и конфигурация провайдеров**  
Полностью скрыто. Пользователь видит результат, не процесс.

**Настройка воронки с нуля**  
Умные defaults из коробки. Пользователь начинает работать, не настраивать.

**Отдельный раздел аналитики в MVP**  
5–7 метрик в контексте: прямо на карточке, прямо в «Сегодня». Отдельный раздел — после первых клиентов, которые его попросят.

## Вопрос 9. Что становится главным

### 5 функций, без которых продукт не существует

Проверка каждой: *«Если убрать — пользователь всё ещё сможет найти клиента и написать ему за 10 минут?»* Для этих пяти ответ «нет».

---

**1. Поиск по намерению**  
Написал что нужно — система нашла. Без фильтров, без технических терминов, без настроек.

*Внутри: Intent Interpreter + Hunt Orchestrator + Plugin Sources*

---

**2. «Что делать сегодня» (только для вернувшихся пользователей)**  
После того как пользователь начал работать, каждый день ему нужен приоритетный список: кому написать первым и почему.

*Внутри: Activity Queue, Signal Engine, AI Scoring*

---

**3. Автоматическое обогащение данных**  
Email, телефон, должность, размер компании — находится само. Пользователь видит готовые данные.

*Внутри: Waterfall Enrichment через Dadata → Hunter → Snov → HH.ru → 2GIS*

---

**4. Ноль ручного ввода после каждого действия**  
После звонка система предлагает заметку и следующий шаг. После письма статус обновился. После встречи появился summary. Пользователь одобряет — не заполняет.

*Внутри: Post-action AI jobs в BullMQ, автогенерация через LLM*

---

**5. Полный цикл в одном месте**  
Сейчас пользователь вынужден собирать стек: Clay + Apollo + Instantly + CRM + мессенджер. Мы закрываем весь путь от поиска до ответа без переключения инструментов.

*Внутри: Unified Channel Abstraction Layer*

---

**Что с Telegram?**  
Telegram — важное конкурентное преимущество для российского рынка. Но это не ядро: если Telegram завтра исчезнет, продукт не должен умереть. Telegram — усилитель, который добавляется поверх функции «Полный цикл в одном месте».

## Вопросы 10–12. Что конкуренты делают неправильно

### Apollo — проблема не данные, а перегрузка

Первый экран Apollo после регистрации — 150 параметров поиска. Человек, который хочет просто найти клиентов, оказывается перед вопросами, на которые не знает ответа.

**Главная проблема Apollo — когнитивная перегрузка.** Не качество данных.

Дополнительно: непрозрачная кредитная система создаёт постоянный anxiety. Плохое качество российских данных. Письма часто попадают в спам.

### Clay — слишком мощный для большинства

Clay — революционный продукт для тех, кто умеет им пользоваться. Waterfall enrichment из 150+ источников. Claygent для AI-ресёрча каждого лида.

Но Clay — инструмент для data-ops специалиста. Без технических знаний пользователь застрянет на настройке.

**Главная проблема Clay — кривая обучения и отсутствие готового результата из коробки.**

### Close — только часть пазла

Лучшая CRM для команд, которые продают по телефону. Chloe AI — один из лучших AI-агентов на рынке. Быстрый онбординг.

Но Close — только CRM. Нет поиска клиентов. Нет обогащения. Нет Telegram. Нет российских данных.

**Главная проблема Close — пользователь должен параллельно использовать ещё 3–4 инструмента.**

## Вопрос 13. Что не сделал никто

**1. Поиск клиентов на естественном языке**  
Везде есть форма с фильтрами. Нигде нет возможности написать «мне нужны рестораны в Казани, которые недавно открылись» и получить список через 2 минуты. Мы можем быть первыми.

**2. Российские источники как встроенная часть**  
2GIS, HH.ru, Dadata, Rusprofile, Kontур.Фокус — ни один западный конкурент их не поддерживает нативно. Огромное белое пятно на карте мирового рынка.

**3. Telegram как канал для работы с клиентами**  
История переписки в Telegram прямо в карточке клиента. В России Telegram — основной канал B2B-коммуникаций. Никто не интегрировал его в систему работы с клиентами.

**4. Продукт для всех, кто продаёт — не только для SDR**  
Все существующие продукты созданы для sales-команд. Юрист, дизайнер, перевозчик, консультант — они тоже ищут клиентов, но ни Apollo, ни Close им не подходят.

**5. Весь цикл без стека инструментов**  
От «описал кого ищу» до «получил ответ» — в одном месте. Без переноса данных между инструментами.

## Вопрос 14. Архитектура

*(Раздел — для технической команды)*

### Что сохраняется из v1.0:
- ✅ Plugin Architecture — ключевое техническое преимущество
- ✅ BullMQ + event-driven — правильно для async AI-операций
- ✅ Multi-tenant + PostgreSQL RLS — foundation для SaaS
- ✅ Drizzle ORM — type-safe, хороший выбор
- ✅ Company как центральная бизнес-сущность
- ✅ Human-in-the-loop принцип

### Что добавляется:

**Intent Interpreter — новый ключевой компонент**  
Принимает текст на естественном языке → превращает в структурированные параметры поиска для Hunt. Без него нет Intent-first подхода.

**Channel Abstraction Layer**  
Email, Telegram, звонки — через единый интерфейс `Channel`. Сейчас только email. Нужен абстрактный слой под все каналы.

**Event Sourcing для истории**  
Все действия — неизменяемые события. История общения — проекция событий. Полная история без потерь.

**AI как сквозной сервис**  
AI не модуль — сервис, доступный из любой части системы: brief перед звонком, summary после, draft письма, приоритизация списка.

**Real-time через WebSocket**  
Обогащение завершено → карточка обновилась. Клиент открыл письмо → уведомление мгновенно.

**Hunt — только в системном слое**  
Таблица `hunts`, API endpoint `/api/v1/hunts`, внутренний класс `HuntOrchestrator`. В пользовательском интерфейсе — никогда.

## Вопрос 20. Ядро AI Sales OS Version 2.0

Продукт существует, если работают эти пять вещей:

| Что пользователь получает | Что работает под капотом |
|--------------------------|-------------------------|
| Нашёл клиентов по описанию | Intent Interpreter + Hunt Orchestrator + Plugin Sources |

## Резюме: v1.0 → v2.0

| Аспект | v1.0 | v2.0 |
|--------|------|------|
| **Для кого** | Sales-команды | Любой, кто продаёт в B2B |
