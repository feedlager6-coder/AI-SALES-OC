## Вопрос 1. Философия AI Sales OS

### Старая философия (v1.0)
> «Мы строим AI Sales CRM с plug-in архитектурой»

### Новая философия (v2.0)
> **«Мы строим AI-мозг, который сам находит клиентов — пользователь только говорит "кого ищу" и подтверждает результаты»**

Три принципа новой философии:

**1. Намерение → Результат, не Инструмент → Настройка**  
Пользователь не настраивает воронку, не выбирает провайдеров обогащения, не строит pipeline вручную. Он описывает намерение: *«Ищу логистические компании в Москве с автопарком 50+ машин»* — система делает всё остальное сама.

**2. AI как невидимый слой, а не отдельная функция**  
AI не должен быть кнопкой «Сгенерировать письмо». AI должен быть в каждом пикселе: подготовил brief перед звонком, предложил статус после разговора, обогатил карточку пока менеджер читал информацию о компании. Пользователь не замечает AI — он просто замечает, что работает быстрее.

**3. Zero Cognitive Load**  
Продукт думает вместо пользователя о ТОМ, ЧТО ДЕЛАТЬ ДАЛЬШЕ. Менеджер по продажам — не аналитик данных. Он не должен думать «какой провайдер использовать для обогащения» или «какой следующий шаг для этого лида». Система знает. Система предлагает. Человек подтверждает.

## Вопрос 2. Что является продуктом?

**Ответ: GTM Intelligence Engine — не CRM, не Apollo, не Clay.**

Разберём по каждому варианту:

| Вариант | Почему не подходит |
|---------|-------------------|
| **CRM** | CRM — это хранилище контактов и сделок. Мы строим систему, которая сама находит кого хранить. CRM — лишь побочный эффект. |
| **AI Workspace** | Слишком размыто. Notion тоже AI Workspace. Нам нужна более конкретная категория. |
| **Sales Copilot** | Copilot — это помощник. Мы строим нечто агрессивнее: систему, которая берёт инициативу, а не ждёт команды. |
| **Search Engine** | Ближе к правде, но не отражает весь lifecycle: поиск → обогащение → приоритизация → персонализация → отправка → трекинг. |
| **GTM Operating System** | ✅ **Это наш ответ.** Операционная система для выхода на рынок. Не просто поиск, а полный цикл от «кого искать» до «отправлено персональное предложение». |

**Но с одним важным уточнением:**  
GTM OS, ориентированная на **российский и СНГ B2B-рынок**, с локальными источниками данных, которые не поддерживает ни один западный продукт.

**Positioning Statement:**
> *AI Sales OS — это единственная GTM Intelligence Platform, спроектированная для B2B-продаж в России и СНГ: от поиска потенциального клиента до персонального предложения — в одном рабочем пространстве, без ручного ввода данных.*

## Вопрос 3. Главная сущность системы

**Ответ: Компания (Company) — остаётся. Но появляется новая суперсущность: Search (Поисковый запрос / Охотничий профиль).**

### Почему Company остаётся главной сущностью:
В B2B продажах деньги платит компания. Контакт (ЛПР) — это точка входа, но решение принимает организация. Это правильное решение, принятое в v1.0, и его не нужно менять.

### Новая суперсущность: **Hunt** (Охота)
В v2.0 добавляется концепция Hunt — сохранённый поисковый профиль с критериями ICP:

```
Hunt {
  id: uuid
  name: "Логистика Москва 2026"
  icp_criteria: {
    industry: ["transport", "logistics"],
    location: ["Москва", "МО"],
    fleet_size: { min: 50 },
    signals: ["открыта вакансия логист", "зарегистрирована < 5 лет"]
  }
  sources: ["2gis", "hhru", "rusprofile"]
  enrichment_waterfall: ["dadata", "hunter", "snov"]
  status: "active" | "paused" | "completed"
  companies_found: 847
  companies_enriched: 312
  companies_contacted: 128
}
```

**Иерархия сущностей v2.0:**
```
Hunt (Охота)
  └── Company (Компания)
       ├── Contact (Контакт / ЛПР)
       ├── Deal (Сделка)
       ├── Timeline (Единая история)
       └── AI Brief (Подготовка к звонку)
```

**Что это меняет в UX:**  
Пользователь начинает не с «Создать компанию», а с «Начать поиск». Это фундаментальный сдвиг точки входа.

## Вопрос 4. Путь пользователя с первой секунды после регистрации

### Целевое время до первой ценности: **< 10 минут**
(Сейчас лучший показатель на рынке: Close — 5-10 мин, Apollo — 15 мин, Clay — 2-4 часа)

### Новый Onboarding Flow v2.0:

```
РЕГИСТРАЦИЯ (30 секунд)
  → Имя + email + пароль
  → Без лишних вопросов «сколько человек в команде»

ЭКРАН 1: Кто ваш идеальный клиент? (2 минуты)
  AI задаёт 3 вопроса голосом / текстом:
  1. «Кому вы продаёте? (отрасль, тип компании)»
  2. «В каком регионе ищете?»
  3. «Что является сигналом, что компания готова купить?»
  
  → Система сама формирует ICP-профиль
  → Показывает предварительную оценку: «Нашёл 2,400 компаний под ваш профиль»

ЭКРАН 2: Первые результаты (3 минуты)
  → 10-20 компаний уже найдено и обогащено
  → Пользователь видит: название, размер, контакт ЛПР, email
  → Кнопка: «Запустить AI-письмо для этих компаний»
  
ЭКРАН 3: Первое письмо отправлено (2 минуты)
  → AI генерирует персональное письмо под каждую компанию
  → Пользователь просматривает 3 примера, одобряет
  → «Запустить для всех 20 компаний»

РЕЗУЛЬТАТ: За 10 минут пользователь нашёл лидов и отправил персональные письма.
Без настройки CRM. Без импорта CSV. Без ручного ввода.
```

**Ключевой принцип:** Пустой экран — это худший опыт. С первой секунды пользователь видит данные, не форму для заполнения.

## Вопрос 6. Ментальная модель пользователя

### Как пользователь должен думать о системе:

**Старая ментальная модель (v1.0):**
> *«Это CRM, где я веду клиентов. Нужно создать компанию, добавить контакт, настроить pipeline, выбрать провайдера обогащения...»*

**Новая ментальная модель (v2.0):**
> *«Это мой AI-сотрудник по поиску клиентов. Я говорю ему кого искать — он находит, обогащает, готовит письмо. Я только проверяю и одобряю.»*

### Три режима работы (которые пользователь понимает интуитивно):

**Режим «Охота» (Hunt Mode)**  
*«Я ищу новых клиентов»*  
Запускаешь поиск → AI находит → Ты выбираешь лучших → Запускаешь коммуникацию

**Режим «Работа» (Work Mode)**  
*«Я работаю с теми, кого уже нашёл»*  
Открываешь Activity Queue → Видишь что делать сегодня → Делаешь → Закрываешь

**Режим «Просмотр» (Review Mode)**  
*«Я смотрю что происходит»*  
Видишь pipeline → Статусы → Что сработало → Что нет

### Чего НЕ должен думать пользователь:
- «Какой провайдер мне выбрать?»
- «Как настроить waterfall?»
- «Куда добавить этот контакт?»
- «Как написать персональное письмо?»

Система отвечает на эти вопросы сама.

## Вопрос 8. Функции, которые перестают быть центральными

### Теряют центральное место:

**1. Ручное создание компаний и контактов**  
В v1.0 это было основным действием. В v2.0 — это крайний случай. 95% компаний и контактов создаются автоматически через Hunt.

**2. Настройка провайдеров обогащения**  
Пользователь не выбирает «использовать Hunter или Snov». Waterfall Enrichment — внутренняя механика. Пользователь видит результат, а не процесс.

**3. Импорт CSV**  
Остаётся как функция, но это уже не «главный способ добавить лидов». Это «запасной вход».

**4. Конфигурация pipeline**  
Pipeline из коробки должен иметь разумные дефолтные стадии. Пользователь не должен начинать с «настройки воронки».

**5. Отдельный раздел аналитики**  
Аналитика переезжает в контекст: метрики прямо на карточке компании, прямо в Activity Queue. Отдельный сложный dashboard — не нужен в MVP.

## Вопрос 9. Функции, которые становятся главными

### Центральные функции v2.0:

**1. 🔮 AI Hunt — умный поиск клиентов**  
Самая важная функция. Пользователь описывает ICP → система автоматически ищет через все подключённые источники (2GIS, HH.ru, Rusprofile, OpenCorporates) → обогащает через waterfall → приоритизирует по ICP Score.

**2. ⚡ Activity Queue с AI-приоритизацией**  
Утром открываешь систему — видишь список: что делать прямо сейчас, в каком порядке, почему. AI ранжирует по сигналам: открыл письмо, отвечает, давно без контакта.

**3. 🤖 Zero Data Entry после каждого действия**  
После звонка → AI предлагает статус, заметку, следующий шаг. После email → статус обновился автоматически. После встречи → создался summary + задачи.

**4. 📊 AI Brief перед каждым контактом**  
Открыл карточку → видишь: «Последнее общение 5 дней назад. Тогда обсуждали стоимость. Он просил перезвонить в пятницу. Новые вакансии: 2 логиста. Рекомендация: напомнить о встрече».

**5. 🌊 Waterfall Enrichment как стандарт**  
Не «выбери провайдера», а «мы нашли email через Hunter (попытка 1), телефон через Dadata (попытка 2), должность через HH.ru (попытка 3)».

**6. 📱 Telegram как первый класс коммуникации**  
Ни один конкурент не поддерживает Telegram как канал CRM. Это killer feature для российского рынка. История переписки в Telegram — прямо в Timeline.

## Вопрос 10. Что Apollo делает неправильно

На основе Product Intelligence Report и анализа отзывов G2 (2025-2026):

**1. Кредитная система создаёт anxiety**  
Apollo продаёт «кредиты» на просмотр контактов. Пользователь постоянно думает «сколько кредитов осталось» вместо «кому позвонить». Это психологический барьер. Решение: flat pricing или неограниченные просмотры.

**2. Перегруженный интерфейс**  
150+ параметров для поиска — это не конкурентное преимущество, это UX-катастрофа для нового пользователя. Мощь должна быть под капотом, а не в интерфейсе.

**3. Deliverability — не их проблема**  
Apollo отправляет письма, но не отвечает за то, что они попадут в inbox. Это толкает пользователей к Instantly. Наша система должна иметь встроенную защиту deliverability.

**4. Нет локальных рынков**  
Apollo — 275M контактов, но почти нет качественных данных по российским компаниям. Это наше прямое окно возможностей.

**5. Data quality нестабильна**  
Устаревшие email, высокий bounce rate. Waterfall enrichment решает эту проблему структурно: проверяем несколько источников, выбираем наиболее свежий.

## Вопрос 11. Что Clay делает неправильно

**1. Требует data-ops специалиста**  
Clay — это инструмент для технарей. Средний SDR не может его настроить. «Spreadsheet с суперсилой» не подходит для команды из 2 человек без RevOps.

**2. Нет CRM, нет sequences**  
Clay отлично обогащает данные, но потом надо всё переливать в другой инструмент. Пользователи вынуждены строить стек: Clay + Instantly + отдельная CRM.

**3. Ценообразование непредсказуемо**  
Data Credits + Actions Credits = два непредсказуемых счётчика. Пользователь не может планировать бюджет.

**4. Слишком горизонтальный продукт**  
Clay делает всё для всех. Мы делаем одно, но лучше всех: B2B продажи для рынка России/СНГ.

**5. Нет локальных источников**  
Clay имеет 150+ провайдеров, но среди них нет 2GIS, HH.ru, Dadata, Rusprofile. Для нашей аудитории это критично.

## Вопрос 12. Что Close делает неправильно

**1. Ориентирован только на inside sales**  
Close — лучший инструмент для команд, которые продают звонками. Но современные B2B продажи — это email + звонок + LinkedIn + Telegram. Close слабее в multichannel.

**2. Нет enrichment и prospecting**  
Close — отличная CRM и диалер, но откуда брать лидов? Пользователь должен использовать Apollo или Clay для поиска, потом переливать в Close.

**3. Реальная цена выше рекламируемой**  
$49/user + стоимость минут диалера + стоимость записи = ~$130/user. Это непрозрачно.

**4. Нет локализации**  
Close полностью ориентирован на западный рынок. Telegram, 2GIS, HH.ru — за пределами их радара.

**5. Chloe AI — революционно, но только для звонков**  
Chloe — лучший AI-агент на рынке CRM (автодозвон, квалификация, обновление CRM). Но только для телефонных продаж. Нам нужен аналог для полного цикла: от поиска до отправки.

## Вопрос 13. Что вообще никто пока не реализовал

**1. 🇷🇺 Российские источники данных как first-class citizens**  
Ни один продукт не поддерживает 2GIS + HH.ru + Dadata + Rusprofile + Kontур.Фокус как встроенные источники. Это белое пятно размером в $20B+ рынок.

**2. Telegram как CRM-канал**  
История переписки в Telegram прямо в timeline лида. Отправка сообщений в Telegram из CRM. Уведомления о событиях в Telegram. Никто не сделал.

**3. Intent Signals из российских источников**  
«Компания открыла вакансию логиста» (HH.ru) = сигнал, что они масштабируются. «Компания зарегистрирована < 2 лет» (ЕГРЮЛ) = сигнал молодого бизнеса. Это информация, которую нельзя получить через западные источники.

**4. AI Hunt — поисковый запрос как основной интерфейс**  
Ни Apollo, ни Clay не дают возможности сказать «найди мне клиентов» в свободной форме с AI-интерпретацией. Везде есть форма с фильтрами. Мы можем сделать первый natural language lead search.

**5. Полный цикл без переключения инструментов**  
Ни один продукт не покрывает весь путь: найти (Hunt) → обогатить (Waterfall) → приоритизировать (AI Score) → написать (AI Composer) → отправить (Sequences) → трекать (Timeline) → работать с ответом (Inbox) — всё в одном окне без интеграций.

**6. Waterfall по российским провайдерам**  
Аналог Clay Waterfall, но для RU рынка: Dadata → Hunter → Snov → Rusprofile → парсинг сайта → LinkedIn. Поиск email и телефона с максимальным покрытием.

## Вопрос 14. Архитектура лучшего AI Sales продукта мира в 2026 году

### Философия архитектуры v2.0:

**Было:** Pipeline-centric (сначала данные → потом AI над ними)  
**Стало:** AI-centric (AI оркестрирует всё, данные появляются как результат)

### Ключевые архитектурные решения:

**1. Hunt Orchestrator — новый центральный компонент**
```
HuntOrchestrator {
  interpretsIntent(text) → ICPProfile    // NLP → структурированный ICP
  plansSources(icp) → SourcePlan         // Какие источники использовать в каком порядке
  executesWaterfall(plan) → Companies[]  // Параллельный/каскадный поиск
  scoresResults(companies) → Ranked[]    // AI-приоритизация
  triggersEnrichment(companies)          // Автоматическое обогащение
}
```

**2. Channel Abstraction Layer — обязательно**
```typescript
interface Channel {
  send(message: OutboundMessage): Promise<SendResult>
  receive(): AsyncIterator<InboundMessage>
  getHistory(contactId: string): Promise<Message[]>
  getStatus(): ChannelStatus
}
// Реализации: EmailChannel, TelegramChannel, CallChannel, LinkedInChannel
```
Сейчас в архитектуре только email через Mailgun. Нужен единый абстрактный слой.

**3. Event Sourcing для Timeline**  
Все события (звонок, письмо, смена статуса, AI-анализ, обогащение) пишутся как неизменяемые события. Timeline — это проекция всех событий. Это обеспечивает:
- Полную историю без потерь
- AI-анализ паттернов («компании, где 3 звонка без ответа — конверсия 2%»)
- Аудит всех действий

**4. AI Layer — сквозной сервис, не модуль**  
AI не должен быть отдельным endpoint. AI встроен в каждый workflow:
- CompanyCard открыта → AI Background Job → Brief готов через 5 секунд
- Call ended → AI Job → Draft summary + next steps за 10 секунд
- Email received → AI Job → Draft reply за 5 секунд
- New company added → AI Job → ICP Score рассчитан за 3 секунды

**5. Intent Signals Aggregator — новый компонент**
```
SignalAggregator {
  monitors: [HHruSignals, DomainSignals, SocialSignals]
  detects: JobPostings, FundingRounds, KeyhireSignals, GrowthSignals
  enriches: Company.signals[] в реальном времени
  triggers: RepriorityAction если сигнал высокий
}
```

**6. Real-time через WebSocket — обязательно**  
- Обогащение завершено → карточка обновилась без перезагрузки
- Лид открыл письмо → уведомление в Activity Queue мгновенно
- AI brief готов → появился в карточке

**7. Сохранить из v1.0 (критически важно):**
- ✅ Plugin Architecture — правильное решение, развивать
- ✅ BullMQ + Event-driven — правильный подход для async AI
- ✅ Drizzle ORM + PostgreSQL — хороший выбор
- ✅ Multi-tenant через workspace_id — правильно
- ✅ Company как основная бизнес-сущность

## Вопрос 20. Ядро AI Sales OS Version 2.0

### Это ядро — 5 элементов, без которых продукт не существует:

---

### 🧠 1. AI Hunt Engine
**Что:** Natural language → ICPProfile → Multi-source Search → AI Scoring → Ready-to-contact list  
**Почему ядро:** Это главная причина прийти в продукт. Если Hunt не работает — нет продукта.  
**Техническая реализация:** HuntOrchestrator service + Plugin sources (2GIS, HH.ru, Rusprofile) + ICP Scorer (LLM-based)  

---

### ⚡ 2. Activity Queue с AI-приоритизацией
**Что:** Каждое утро — персональная очередь задач с объяснением приоритета  
**Почему ядро:** Это причина возвращаться в продукт каждый день. Habit-forming feature.  
**Техническая реализация:** Signal Aggregator + Priority Algorithm (открытия писем, время без контакта, сигналы покупки) + Queue Builder  

---

### 🌊 3. Waterfall Enrichment для RU рынка
**Что:** Автоматическое обогащение из каскада российских источников (Dadata → Hunter → Snov → HH.ru → 2GIS → сайт)  
**Почему ядро:** Это наше техническое преимущество перед всеми конкурентами. Никто не сделал waterfall специально для RU источников.  
**Техническая реализация:** Уже заложена в plugin system. Нужно добавить 2GIS, HH.ru, Rusprofile плагины.  

---

### 🤖 4. Zero Data Entry AI Layer
**Что:** После каждого действия AI автоматически предлагает заметку, статус, следующий шаг. Пользователь только одобряет.  
**Почему ядро:** Это то, что сделает систему незаменимой. Менеджеры ненавидят data entry — мы убираем его полностью.  
**Техническая реализация:** Post-action AI jobs в BullMQ + LLM для summarization + UI для quick approve/edit  

---

### 📱 5. Telegram-first коммуникации
**Что:** Telegram как полноценный CRM-канал: история переписки в Timeline, отправка из CRM, уведомления менеджеру  
**Почему ядро:** Это killer feature для российского рынка. Никто не сделал. Telegram — основной канал B2B коммуникаций в РФ.  
**Техническая реализация:** Telegram Bot API + Channel Abstraction Layer + TelegramChannel implementation
